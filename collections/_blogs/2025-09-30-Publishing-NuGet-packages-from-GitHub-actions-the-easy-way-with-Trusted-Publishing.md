Andrew Lock walks through using nuget.org’s Trusted Publishing to publish NuGet packages from a GitHub Actions workflow without storing long-lived API keys, using OIDC token exchange and a nuget.org trust policy to improve publishing security.<!--excerpt_end-->

## Overview

nuget.org now supports **Trusted Publishing**, which lets you publish NuGet packages from CI (for example, **GitHub Actions**) without generating, storing, and rotating long-lived **NuGet API keys**. Instead, the workflow uses **OpenID Connect (OIDC)** to obtain a short-lived token that nuget.org can exchange for a short-lived API token.

## How people typically push NuGet packages

Common approaches include:

- Build locally and manually upload to nuget.org.
- Build in CI, download artifacts, and manually upload.
- Build in CI and push directly to nuget.org from the pipeline.

Pushing from CI usually requires:

- Generating a NuGet API key
- Storing it as a GitHub Actions secret
- Passing it into the workflow
- Rotating it as needed
- Managing access for teams/orgs

This increases operational risk because **long-lived secrets** are easy to leak and hard to manage.

## What Trusted Publishing is

Trusted Publishing connects CI systems and package repositories using standard auth flows (OIDC), similar to what other ecosystems do (PyPI, RubyGems, npm).

For **GitHub Actions + nuget.org**, the flow is:

- Configure a **trust policy** on nuget.org.
- In the GitHub Actions workflow, request an **OIDC token** (GitHub is the IdP).
- The token is a signed **JWT** containing repo/workflow identity details.
- Send the token to nuget.org to exchange it.
- nuget.org verifies the JWT against the configured trust policy.
- nuget.org issues a **short-lived API token**.
- Use that token to publish the `.nupkg` to nuget.org.

## Configuring Trusted Publishing on nuget.org

The post demonstrates adding publishing to an existing workflow for the author’s `sleep-pc` package.

### Starting workflow (build and pack only)

Stored as `.github/workflows/BuildAndPack.yml`:

```yml
name: BuildAndPack
on:
  push:
    branches: [main]
    tags: ['*']
  pull_request:

jobs:
  build-and-publish:
    permissions:
      contents: read
    runs-on: windows-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-dotnet@v4
        with:
          dotnet-version: 10.0.x

      - name: dotnet pack
        run: |
          dotnet pack
          dotnet pack -r win-x64
```

Note: `permissions: contents: read` is used to limit the default `GITHUB_TOKEN` permissions.

### Update the workflow for Trusted Publishing

Changes needed:

- Add `id-token: write` so GitHub can mint an OIDC token for the job.
- Use `NuGet/login@v1` to exchange the OIDC token for a NuGet API key.
- Push the packages using the generated token.

Updated workflow (as shown in the post):

```yml
name: BuildAndPack
on:
  push:
    branches: [main]
    tags: ['*']
  pull_request:

jobs:
  build-and-publish:
    permissions:
      contents: read
      id-token: write # enable GitHub OIDC token issuance for this job

    runs-on: windows-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-dotnet@v4
        with:
          dotnet-version: 10.0.x

      - name: dotnet pack
        run: |
          dotnet pack
          dotnet pack -r win-x64

      # Use the ambient GitHub token to login to NuGet and retrieve an API key
      - name: NuGet login (OIDC → temp API key)
        uses: NuGet/login@v1
        id: login
        with:
          # Secret is your NuGet username, e.g. andrewlock
          user: ${{ secrets.NUGET_USER }}

      - name: push to NuGet
        # Only push to NuGet if we're building a tag (optional)
        if: startsWith(github.ref, 'refs/tags/')
        shell: pwsh
        # Loop through all the packages in the output folder and push them to
        # nuget.org, using the NUGET_API_KEY generated by the previous login step
        run: |
          Get-ChildItem artifacts/package/release -Filter *.nupkg | ForEach-Object {
            dotnet nuget push $_.FullName `
              --api-key "${{ steps.login.outputs.NUGET_API_KEY }}" `
              --source https://api.nuget.org/v3/index.json
          }
```

The only secret the workflow needs is `NUGET_USER` (your nuget.org username).

### Configure a trust policy on nuget.org

If you run the workflow before creating the trust policy, the `NuGet login` step fails with an error like:

```text
Requesting GitHub OIDC token from: https://run-actions-1-azure-eastus.actions.githubusercontent.com/...
Error: Token exchange failed (401): No matching trust policy owned by user '***' was found.
```

To fix this, create a trust policy on nuget.org:

- Sign in to nuget.org
- Go to the Trusted Publishing page: https://www.nuget.org/account/trustedpublishing
- Click **Create** and fill in:
  - Policy name (often repo/package name)
  - Package owner (user/org)
  - GitHub repo owner + repository
  - Workflow file that will publish

![The Trusted Publishing page is available in the account menu](/content/images/2025/trusted_publishing_02.png)

![The configured details for the sleep-pc tool](/content/images/2025/trusted_publishing_03.png)

After creation, the policy is partially active until used:

![The partially active policy](/content/images/2025/trusted_publishing_04.png)

Re-run the workflow; a successful exchange looks like:

```text
Requesting GitHub OIDC token from: https://run-actions-1-azure-eastus.actions.githubusercontent.com/...
Successfully exchanged OIDC token for NuGet API key.
```

Then the policy becomes fully active:

![The fully active policy](/content/images/2025/trusted_publishing_05.png)

## Wrap-up

The main benefits described are:

- Easier CI publishing to nuget.org
- Avoiding long-lived credentials (reduced secret-management burden)

The author notes potential future ecosystem improvements (for example, verified marks and possible ties to provenance attestations), but today the key win is removing long-lived API keys from CI.
